# Implementation Plan: Completion Instruction

**Branch**: `003-new-instruction-complete` | **Date**: 2026-04-16 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/003-new-instruction-complete/spec.md`

**Note**: This plan is filled in by the native `/spec plan` workflow.

## Summary

Add a new AgeRun language instruction, `complete(...)`, that fills interpolated `memory...`
variables from a local CPU-only completion backend. Per the latest planning constraint, all new
feature-specific modules for this work will be implemented as Zig C-ABI modules with stable C
headers: a dedicated Zig parser module, a dedicated Zig evaluator module, and a dedicated Zig
`ar_local_completion` adapter that hides a pinned, vendored, CPU-only `llama.cpp` / `libllama`
integration plus the `phi-3-mini-q4.gguf` model asset. Existing facade modules such as
`ar_instruction_parser.c` and `ar_instruction_evaluator.zig` will be updated only to dispatch into
those new modules. The evaluator will derive placeholder targets from the template, require
placeholder names to follow the same identifier syntax already used by `build(...)` and `parse(...)`,
request structured placeholder values from the local completion backend, validate full coverage,
require text generated by the chosen local language model, reject empty values, leading/trailing
whitespace, `{` or `}` characters, and invalid placeholder names, reject invalid requests before
local generation begins, treat missing or unusable local completion resources on otherwise supported
macOS/Linux baseline environments as runtime failures rather than portability exclusions, keep
partial-generation failures under the same waiting-limit and failure contract, apply the same
template-interpretation rules for punctuation and quoted literals across supported environments,
preserve caller-provided values as literals without treating their brace characters as generated
markers, and return a new map rather than mutating the provided values map.

## Technical Context

**Language/Version**: Existing AgeRun C17 runtime with Zig 0.14.1 C-ABI support; all new feature-specific modules for this feature are Zig modules
**Primary Dependencies**: `ar_instruction_parser`, `ar_instruction_evaluator`, `ar_instruction_ast`, `ar_expression_evaluator`, `ar_frame`, `ar_data`, `ar_log`, `ar_path`, `ar_io`, Zig 0.14.1, Make, pinned vendored `llama.cpp` / `libllama` C API, local `phi-3-mini-q4.gguf` model asset
**Storage**: In-memory agent memory updates plus read-only local model file access from disk; no network storage and no persistent completion cache in the first release
**Testing**: `make ar_complete_instruction_parser_tests 2>&1`, `make ar_complete_instruction_evaluator_tests 2>&1`, `make ar_local_completion_tests 2>&1`, `make ar_instruction_parser_tests 2>&1`, `make ar_instruction_ast_tests 2>&1`, `make ar_instruction_evaluator_tests 2>&1`, `make sanitize-tests 2>&1`, `make check-docs`, `make clean build 2>&1`, `make check-logs`
**Target Platform**: macOS and Linux local CLI/runtime environments with CPU-only model execution, with first-release support and validation tied to a documented minimum hardware baseline per operating system; portability guarantees are baseline-based rather than instruction-set-specific and require no GPU, NPU, or other hardware accelerator for normal supported operation
**Project Type**: Message-driven runtime language feature spanning parser, evaluator, local backend adapter, and documentation/contracts
**Performance Goals**: Match [spec.md](./spec.md) SC-001, SC-001a, SC-002, SC-002a, SC-002b, SC-002c, SC-002d, SC-002e, SC-002f, SC-002g, SC-003b, SC-003c, SC-003d, and SC-004 by validating documented acceptance and failure fixture sets, requiring at least 90% warm-run completion within 15 seconds after runtime initialization and before the runtime begins processing the next queued message, at least 90% cold-start completion within 30 seconds including first model load under the same next-queued-message boundary, the same latency target for repeated-placeholder short templates, invalid-before-generation failure without local completion invocation, timeout classification whenever the configured waiting limit is exceeded, 100% required actionable-error fields across all calls in the documented failure fixture set, and success-only acceptance when missing placeholder values are text generated by the chosen local language model and caller-provided values are preserved as copied literals. First-release performance guarantees apply only when no other `complete(...)` evaluation is active; below-baseline or heavier-than-documented validation conditions keep failure-safety guarantees but do not retain SC-002/SC-002a timing guarantees.
**Constraints**: zero memory leaks; make-target-only validation; local-only backend with no required network service; `complete(...)` writes strings into `memory...` targets and returns boolean status only; optional second argument limited to direct `memory...` access paths; valid direct `memory...` paths are language-defined and not environment-config-dependent on supported environments; placeholder names must follow the same identifier syntax already used by `build(...)` and `parse(...)`; all new feature-specific modules must be implemented as Zig C-ABI modules with stable `.h` headers; existing facade modules may be updated for dispatch only; the direct backend path must not depend on Homebrew or any other machine-specific system package assumption and should link against a project-controlled vendored CPU-only `libllama` build; on otherwise supported environments, missing or unusable local completion resources are runtime failures rather than unsupported-platform cases; portability includes the documented resource-discovery contract but not broader packaging/distribution guarantees; validation must include documented macOS and Linux evidence for the first release, including the minimum hardware baseline used for each operating system; update `SPEC.md`, module docs, and instruction reference docs together
**Scale/Scope**: First release supports one template string with one or more `{placeholder}` variables, one optional base direct `memory...` access path, one local completion request evaluated synchronously per instruction call, string-only populated values, one shared local model runtime per process loaded lazily and reused across calls, measurable performance guarantees only for short templates as defined in the spec, equal performance/portability treatment across one-argument vs two-argument forms and top-level vs nested writes within that short-template class, portability/performance acceptance only on environments that meet or exceed the documented per-OS minimum hardware baseline, and first-release guarantees only when at most one `complete(...)` evaluation is active at a time

**Structure Decision**: Implement `complete(...)` with two specialized Zig C-ABI modules for
parsing/evaluation ‚Äî `ar_complete_instruction_parser.zig` and
`ar_complete_instruction_evaluator.zig` ‚Äî plus a stable C-header local runtime adapter module,
`ar_local_completion.c`, for the direct `llama.cpp` / `libllama` integration path. This C
implementation is a user-approved exception to the earlier Zig-only assumption because dynamic
loading and low-level `llama.cpp` interop are materially simpler there. Existing facade modules
continue to own top-level dispatch, so `ar_instruction_parser.c` wires the new parser module and
`ar_instruction_evaluator.zig` wires the new evaluator module without introducing additional public
facade layers. The feature updates the public language contract in `SPEC.md` and the instruction
reference docs, but does not require a new built-in method or shell feature to expose it.
