# Feature Specification: Completion Instruction

**Feature Branch**: `003-new-instruction-complete`
**Created**: 2026-04-16
**Status**: Draft
**Input**: User description: "new instruction `complete(\"The largest country in South America is {country}.\")` - it should be implemened using lama.cpp + phi-3-mini-q4.gguf running on CPU."

## Clarifications

### Session 2026-04-16

- Q: How are interpolated variables mapped to memory targets? → A: With one argument, `{country}` targets top-level `memory.country`; with a second argument such as `memory.location`, `{country}` targets `memory.location.country`.
- Q: What should `complete(...)` return? → A: A boolean success/failure result.
- Q: What should happen if target variables already have values? → A: On success, `complete(...)` overwrites all targeted variables atomically.
- Q: What type should populated values use? → A: All populated values are stored as strings.
- Q: What kinds of base paths are allowed for the optional second argument? → A: Only direct `memory...` access paths are allowed, because `context` is read-only.
- Q: How should performance timing be measured? → A: Define separate cold-start and warm-run thresholds.
- Q: What cold-start threshold should apply? → A: Cold-start completion must finish within 30 seconds including first model load.
- Q: Which platforms are in scope for the first release? → A: The first release must support both macOS and Linux.
- Q: Are empty generated values allowed on success? → A: No; any empty generated value makes the instruction fail.
- Q: What qualifies as a short completion template? → A: A template with up to 2 placeholders and up to 120 total characters.
- Q: What does “fits the template” mean? → A: The placeholder targets must be populated with text generated by the chosen local language model, and substituting those values into the original template must preserve every literal segment exactly; each generated value must be a non-empty string with no leading/trailing whitespace and no `{` or `}` characters.
- Q: What does “supported local runtime profile” mean? → A: It means a documented minimum hardware baseline per OS that first-release support must meet or exceed.
- Q: What does the performance timer include? → A: It includes the full instruction time: generation, validation, error reporting, and atomic memory writes.
- Q: What qualifies as a valid optional second argument? → A: Only direct `memory...` access paths such as `memory.city` or `memory.location.city` are valid.
- Q: What minimum detail makes an error actionable? → A: It must include failure category, cause, and recovery hint.
- Q: Which interpolation syntax is supported in the first release? → A: Only `{name}` interpolation is supported in the first release; all other interpolation syntax is out of scope.
- Q: What placeholder-name syntax should `complete(...)` use? → A: It uses the same identifier syntax as `build(...)` and `parse(...)`: ASCII letter first, then ASCII letters, digits, or `_`.
- Q: If local completion resources are missing or unusable on a supported environment, is that an unsupported platform or a runtime failure case? → A: It is a runtime failure case on an otherwise supported environment.
- Q: What does “block message handling” mean for `complete(...)`? → A: It means the runtime cannot begin processing the next queued message until the current `complete(...)` call succeeds or fails.
- Q: What concurrency assumption applies to first-release `complete(...)` performance and portability expectations? → A: They assume at most one active `complete(...)` evaluation at a time, with no concurrency guarantee.
- Q: How should SC-003b and SC-004 avoid reviewer-selected subsets? → A: They apply to all calls in the documented validation or acceptance fixture set, not a reviewer-selected subset.
- Q: Do first-release performance and portability guarantees apply equally across one-argument vs two-argument calls and top-level vs nested targets? → A: Yes. They apply equally as long as the template still qualifies as a short completion template.
- Q: What exactly counts as an “over-limit evaluation”? → A: It means the full `complete(...)` instruction path exceeded the configured waiting limit before success or failure completed.
- Q: Are throughput and concurrent `complete(...)` performance part of first-release guarantees? → A: No. First-release guarantees cover only per-call latency on the documented single-active-evaluation path.
- Q: Does first-release portability include packaging/distribution guarantees, or only runtime behavior plus resource discovery? → A: It includes runtime behavior plus the documented resource-discovery contract, but not broader packaging or distribution guarantees.
- Q: What does “CPU-only runtime profile” mean for first-release support? → A: It means normal supported operation requires no GPU, NPU, or other hardware accelerator.
- Q: Can a valid direct `memory...` target become invalid because of environment-specific configuration? → A: No. Direct `memory...` path validity is determined by the language contract, not by environment-specific configuration.
- Q: How do invalid-before-generation, partial-generation, and repeated-placeholder cases affect first-release performance guarantees? → A: Invalid-before-generation requests fail without invoking local completion resources, partial-generation failures remain subject to the same waiting limit, and repeated placeholders within a short template remain under the same latency target.
- Q: How do slower-than-baseline environments or heavier local-runtime conditions affect first-release guarantees? → A: Failure-safety guarantees still apply, but the SC-002 and SC-002a performance guarantees do not.
- Q: Must supported environments interpret punctuation, quoted literals, and repeated placeholders consistently? → A: Yes. Supported environments use the same template-interpretation and failure rules for those cases.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Populate memory from a completion template (Priority: P1)

A method author can submit a sentence template containing interpolated variables and have AgeRun
populate those memory variables with generated values that fit the template.

**Why this priority**: This is the core feature behavior. If the interpolated memory variables are
not populated correctly, the new instruction has no practical value.

**Independent Test**: Execute a method that calls
`complete("The largest country in South America is {country}.")` and verify that the targeted
memory variable for `country` is populated with text generated by the chosen local language model
and that substituting the stored value back into the template preserves the original literal text.

**Acceptance Scenarios**:

1. **Given** agent memory does not yet contain a value for `country`, **When** a method evaluates `complete("The largest country in South America is {country}.")`, **Then** the top-level memory variable `memory.country` is populated with a non-empty generated value.
2. **Given** a method evaluates `complete("The capital is {city}.", memory.location)`, **When** the instruction succeeds, **Then** the nested memory variable `memory.location.city` is populated.
3. **Given** targeted variables already contain values, **When** `complete(...)` succeeds, **Then** it overwrites all referenced target variables as one atomic update.
4. **Given** a completion template contains multiple interpolated variables, **When** the instruction succeeds, **Then** every referenced memory variable is populated during the same completion attempt.

---

### User Story 2 - Reuse populated variables in later method logic (Priority: P2)

A method author can use the memory variables populated by `complete(...)` in later assignments,
messages, or string-building instructions without extra translation steps.

**Why this priority**: The instruction is only useful if its outputs plug directly into existing
AgeRun method workflows.

**Independent Test**: Execute a method that calls `complete(...)` and then uses the populated memory
variables in later instructions, verifying that downstream instructions can read and use those
values normally.

**Acceptance Scenarios**:

1. **Given** `complete(...)` has populated a memory variable, **When** a later instruction reads that variable, **Then** the value is available as normal method memory.
2. **Given** the same interpolated variable appears multiple times in one template, **When** the instruction succeeds, **Then** all references resolve to one consistent stored value for that variable.

---

### User Story 3 - Fail safely when completion is unavailable (Priority: P3)

A runtime operator can continue using the system even when local completion cannot produce values,
without crashes, corrupted memory, or unclear failure behavior.

**Why this priority**: Local completion introduces a new runtime dependency, so failure behavior
must be predictable and safe.

**Independent Test**: Run a method using `complete(...)` while the local completion capability is
unavailable or cannot answer, and verify that the runtime reports failure, preserves runtime
stability, and leaves previously stored values unchanged.

**Acceptance Scenarios**:

1. **Given** a method evaluates `complete(...)` and the local completion capability is unavailable, **When** the instruction is evaluated, **Then** the runtime reports the documented failure status, records an actionable error, and keeps the agent runtime operational.
2. **Given** one `complete(...)` call fails, **When** the same agent later handles another message, **Then** normal non-completion instructions still work and previously stored target values are not overwritten by partial failure output.

- What happens when the local completion capability cannot be initialized or the local completion asset is missing?

## Scope Boundaries *(mandatory)*

### In Scope

- A new AgeRun language instruction named `complete(...)`
- Using one template string as the required instruction input
- An optional second argument that supplies the base memory path for interpolation targets
- Restricting that optional second argument to direct `memory...` access paths only
- Templates that contain one or more interpolated variables written with `{name}` syntax
- First-release support only for `{name}` interpolation syntax
- Populating the memory variables referenced by those interpolations
- Reusing populated memory variables in later instructions
- Safe local execution without requiring an external online service
- First-release portability of runtime behavior plus the documented resource-discovery contract for local completion resources
- Support for environments where only CPU execution is available, meaning no GPU, NPU, or other hardware accelerator is required for normal supported operation
- First-release support for both macOS and Linux local runtime environments
- Predictable failure handling when local completion cannot produce all required values

### Out of Scope

- Returning the fully completed sentence as the primary output of the instruction
- Multi-turn chat sessions or conversational memory inside `complete(...)`
- Streaming partial output back to agents while generation is still running
- Remote hosted completion services as a required dependency
- Fine-tuning, training, or user-managed model customization workflows
- Broader packaging or distribution guarantees for local completion resources beyond the documented resource-discovery contract
- A first-release promise of arbitrary long-form content generation
- Performance guarantees for templates longer than 120 characters or containing more than 2 placeholders
- Throughput guarantees or concurrent `complete(...)` performance guarantees in the first release
- Performance guarantees on environments below the documented minimum hardware baseline or under heavier local-runtime conditions than the documented validation procedure
- Literal-brace escaping rules beyond the documented interpolation format unless added in a later specification
- Interpolation syntaxes other than `{name}` unless added in a later specification

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST provide a new AgeRun language instruction named `complete(...)`.
- **FR-002**: `complete(...)` MUST accept one required string argument representing a completion template.
- **FR-002a**: `complete(...)` MAY accept a second argument specifying the base memory path to use for interpolation targets.
- **FR-002b**: If a second argument is provided, it MUST be a direct `memory...` access path such as `memory.city` or `memory.location.city`.
- **FR-002c**: A direct `memory...` access path that is valid under the language contract MUST be treated as valid consistently across supported environments and MUST NOT become invalid solely because of environment-specific configuration.
- **FR-003**: A completion template MUST support one or more interpolated variables written with `{variable_name}` syntax.
- **FR-003a**: In the first release, `{variable_name}` is the only supported interpolation syntax.
- **FR-003b**: In the first release, each placeholder name inside `{...}` MUST follow the same identifier syntax used by `build(...)` and `parse(...)`: an ASCII letter first, followed by zero or more ASCII letters, digits, or `_`.
- **FR-004**: When `complete(...)` succeeds, it MUST populate the memory variable identified by each interpolation with text generated by the chosen local language model.
- **FR-004c**: For first-release acceptance, generated values fit the submitted template only if substituting them into the original template preserves every literal segment exactly and each generated value is a non-empty string with no leading or trailing whitespace and no `{` or `}` characters.
- **FR-004a**: If no second argument is provided, `{country}`-style interpolations MUST target top-level memory keys such as `memory.country`.
- **FR-004b**: If a second argument is provided, `{country}`-style interpolations MUST target child keys under that base memory path, so `complete(..., memory.location)` stores `{country}` in `memory.location.country`.
- **FR-004d**: First-release performance and portability guarantees apply equally to one-argument and two-argument `complete(...)` calls, and equally to top-level and nested target writes, provided the template still qualifies as a short completion template.
- **FR-004e**: Across supported environments, `complete(...)` MUST apply the same template-interpretation rules for punctuation and quoted literal segments.
- **FR-005**: Populated values MUST be stored as normal agent memory so later AgeRun instructions can reuse them without additional conversion.
- **FR-005a**: All values populated by `complete(...)` MUST be stored as strings in the first release.
- **FR-006**: If the same interpolated variable appears more than once in one template, all references MUST resolve to one consistent stored value for that variable.
- **FR-006a**: Repeated placeholders in a short completion template remain subject to the same first-release performance and portability guarantees as other short completion templates.
- **FR-007**: The instruction's completion output MUST be written to the interpolated memory variables rather than returned as the completed sentence in the instruction result.
- **FR-008**: If the instruction result is observed or assigned, it MUST return a boolean success/failure value rather than the completed sentence text.
- **FR-009**: The feature MUST operate entirely from locally available completion resources and MUST NOT require a remote network service for normal use.
- **FR-009a**: First-release portability requirements include the documented local resource-discovery contract for local completion resources, but do not include broader packaging or distribution guarantees.
- **FR-010**: The feature MUST support the project's CPU-only runtime profile.
- **FR-010c**: For first-release support, a CPU-only runtime profile means normal supported operation requires no GPU, NPU, or other hardware accelerator.
- **FR-010d**: First-release portability guarantees are baseline-based rather than instruction-set-specific: any CPU architecture on supported macOS or Linux that meets the documented minimum hardware baseline and CPU-only definition is in scope.
- **FR-010a**: The first release MUST support both macOS and Linux local runtime environments.
- **FR-010b**: First-release performance and portability guarantees apply only to macOS and Linux environments that meet or exceed the documented minimum hardware baseline for that operating system.
- **FR-011**: If local completion cannot be initialized, cannot produce all required values, or exceeds the supported waiting limit, `complete(...)` MUST report failure status.
- **FR-011a**: On an operating system and hardware baseline that are otherwise supported for the first release, missing or unusable local completion resources MUST be treated as a runtime failure case rather than as proof that the environment itself is unsupported.
- **FR-011b**: If a failure occurs after some placeholder values have been produced but before the call completes successfully, the call remains subject to the same waiting limit and failure contract as any other `complete(...)` evaluation.
- **FR-011c**: On supported environments, environment-specific differences in local completion resource behavior that prevent completion MUST be handled under the same runtime-failure contract rather than as separate portability semantics.
- **FR-012**: When `complete(...)` reports failure, the runtime MUST record an actionable error through the existing error-reporting path.
- **FR-012a**: For first-release acceptance, an actionable error MUST include the failure category, the immediate cause, and a recovery hint.
- **FR-013**: A failed `complete(...)` evaluation MUST NOT crash the runtime, corrupt agent memory, or prevent later message processing.
- **FR-013a**: After `complete(...)` reports timeout or any other failure, the runtime MUST be able to begin the next queued message according to normal scheduling with no additional cooldown imposed by the failed call.
- **FR-014**: If a template contains no interpolated variables, the runtime MUST reject the request as invalid, report failure status, and record an actionable error.
- **FR-014a**: If the second argument is not a valid direct `memory...` access path, the runtime MUST reject the request as invalid, report failure status, and record an actionable error.
- **FR-014b**: If a `complete(...)` request is invalid before local generation begins, the runtime MUST reject it without invoking local completion resources.
- **FR-015**: Successful completion MUST populate all interpolated variables referenced by the template during one completion attempt; otherwise the instruction MUST fail.
- **FR-015c**: If any generated placeholder value is an empty string, the instruction MUST fail.
- **FR-015a**: On success, `complete(...)` MUST overwrite any existing values at all referenced target variables.
- **FR-015b**: All target-variable updates from one successful `complete(...)` call MUST be applied atomically.
- **FR-016**: If `complete(...)` fails, it MUST NOT overwrite already stored target variables with partial or incomplete output.
- **FR-017**: Generated values stored on success MUST NOT contain unresolved interpolation markers from the submitted template.
- **FR-017a**: Generated values stored on success MUST be non-empty strings.
- **FR-018**: The runtime MUST bound how long one `complete(...)` evaluation may block message handling and MUST treat an over-limit evaluation as a failure.
- **FR-018d**: An over-limit evaluation occurs when the full `complete(...)` instruction path exceeds the configured waiting limit before success or failure completes.
- **FR-018e**: The same waiting-limit rule applies whether failure happens before any placeholder value is produced or after partial generation has already occurred.
- **FR-018c**: For `complete(...)`, blocking message handling means the runtime cannot begin processing the next queued message until the current `complete(...)` call succeeds or fails.
- **FR-018b**: For first-release performance guarantees, completion timing includes the full instruction path: generation, validation, error reporting, and atomic memory writes.
- **FR-018a**: For first-release performance guarantees, a short completion template is defined as a template containing up to 2 placeholders and up to 120 total characters.

### Key Entities *(include if feature involves data)*

- **Completion Template**: A string provided to `complete(...)` that combines literal text with one or more interpolated variables indicating which memory values should be generated.
- **Base Memory Path**: An optional direct `memory...` access path argument that redirects interpolation targets from top-level agent memory into a nested memory location.
- **Interpolated Memory Variable**: A memory variable named by an interpolation in the template and populated or overwritten by a successful `complete(...)` call as a string value.
- **Completion Status**: The boolean success-or-failure result of a `complete(...)` evaluation, separate from the generated memory values.
- **Local Completion Resource**: The locally available language-generation capability that AgeRun uses to satisfy `complete(...)` requests without requiring an online service.

## Assumptions & Dependencies *(mandatory)*

- **Assumptions**:
  - The first release of `complete(...)` is template-oriented rather than open-ended free-form continuation.
  - With one argument, `{country}` identifies the top-level memory variable `memory.country`.
  - With a second argument such as `memory.location`, `{country}` identifies the nested target `memory.location.country`.
  - The optional second argument is limited to direct `memory...` access paths because `context` is read-only and cannot be used as a completion target.
  - The completed sentence itself is not the instruction's primary output; populated memory variables are the primary output.
  - If the instruction result is assigned, it is used only for boolean completion status.
  - Successful `complete(...)` outputs are stored as strings rather than inferred numeric or structured types in the first release.
  - Warm-run timing is measured separately from cold-start timing in the first release.
  - First-release performance and portability expectations assume at most one active `complete(...)` evaluation at a time; no concurrency guarantee is made.
  - First-release guarantees cover per-call latency only and do not include throughput or concurrent `complete(...)` performance guarantees.
  - A direct `memory...` path that is valid under the language contract does not become invalid because of environment-specific configuration on supported environments.
  - Cold-start timing includes the first local model load and uses a 30-second threshold in the first release.
  - On environments below the documented minimum hardware baseline or under heavier local-runtime conditions than the documented validation procedure, failure-safety guarantees remain in force but SC-002 and SC-002a performance guarantees do not.
  - The supported local runtime profile is defined by a documented minimum hardware baseline per OS rather than any arbitrary CPU-only machine.
  - The feature is intended for short factual or structured sentence completion rather than long narrative generation.
  - In the first release, a short completion template means up to 2 placeholders and up to 120 total characters.
- **External Dependencies**:
  - A locally available completion runtime packaged or configured for the project
  - A locally available completion asset compatible with CPU-only execution
- **Assumption Clarification**:
  - On an otherwise supported macOS or Linux environment that meets the documented minimum hardware baseline, missing or unusable local completion resources are treated as runtime failure conditions rather than as portability exclusions.
- **Open Questions**: None

## Documentation & Compatibility Impact *(mandatory)*

- **Affected Documentation**: SPEC.md, language grammar documentation, module documentation for parsing/evaluation, method documentation that demonstrates language features, and README usage examples if the new instruction is user-facing
- **Affected Runtime Contracts**: AgeRun instruction grammar, instruction parsing, instruction evaluation, method authoring guidance, and any shell or CLI surfaces that expose language help
- **Compatibility Notes**: This is an additive language feature. Existing instructions should continue to behave unchanged.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: In acceptance testing with 20 short completion templates, at least 18 successful results populate every referenced memory variable with non-empty text generated by the chosen local language model such that substituting the stored values into the original template preserves every literal segment exactly.
- **SC-001a**: In validation scenarios where any generated placeholder value would otherwise be empty, 100% of `complete(...)` calls fail rather than storing an empty string.
- **SC-002**: On the supported local runtime profile, at least 90% of short completion templates finish within 15 seconds after the local completion runtime is already initialized, measured across the full instruction path including generation, validation, error reporting, and atomic memory writes, before the runtime begins processing the next queued message.
- **SC-002e**: Any `complete(...)` call that exceeds the configured waiting limit before success or failure completes is counted as a timeout failure rather than as a successful completion.
- **SC-002a**: On the supported local runtime profile, at least 90% of short completion templates finish within 30 seconds when measured from a cold start that includes the first local model load and the full instruction path.
- **SC-002b**: For SC-001, SC-002, and SC-002a, a short completion template means a template with up to 2 placeholders and up to 120 total characters.
- **SC-002c**: SC-002 and SC-002a apply only when no other `complete(...)` evaluation is active.
- **SC-002d**: SC-002 and SC-002a apply equally to one-argument and two-argument `complete(...)` calls and to top-level and nested target writes, provided the template still qualifies as a short completion template.
- **SC-002f**: Within the documented short-template class, repeated-placeholder use remains subject to the same latency targets as other short completion templates.
- **SC-002g**: In validation scenarios where a `complete(...)` request is invalid before local generation begins, 100% of such calls fail without invoking local completion resources and before the runtime begins processing the next queued message.
- **SC-003**: In validation scenarios where the local completion resource is unavailable, 100% of `complete(...)` calls report failure without crashing the runtime or altering previously stored target values.
- **SC-003b**: In all failure cases included in the documented validation fixture set, 100% of recorded actionable errors include failure category, cause, and recovery hint.
- **SC-003a**: Validation coverage for the first release includes both macOS and Linux local runtime environments that meet or exceed the documented minimum hardware baseline for that operating system.
- **SC-003c**: On supported environments that fall below the documented minimum hardware baseline or run under heavier local-runtime conditions than the documented validation procedure, failure-safety requirements still apply, but SC-002 and SC-002a performance guarantees do not.
- **SC-003d**: In timeout-failure scenarios, 100% of `complete(...)` calls allow the next queued message to begin on the next normal scheduling opportunity after failure is reported.
- **SC-004**: In all successful `complete(...)` calls included in the documented acceptance fixture set, 100% produce string memory values that later instructions can read and reuse without extra conversion logic.
- **SC-005**: In validation scenarios involving pre-existing target values and multiple interpolations, 100% of successful `complete(...)` calls replace all referenced target values together, with no partial updates observed.
